6. Cách tương tác với Remote Repository

Tìm hiểu Remote Repository, git remote, git fetch, git pull, git push và cách nhiều lập trình viên đồng bộ mã nguồn qua một kho trung tâm.

Giới thiệu

Ở các bài trước, bạn đã làm việc với Git hoàn toàn trên máy cá nhân: tạo repository, commit, xem log, cấu hình .gitignore. Nhưng version control chỉ chạy trên một máy thì không đủ khi nhiều người cùng làm chung một dự án. Bài này giới thiệu Remote Repository — kho trung tâm giúp nhiều lập trình viên cùng đẩy (push) và kéo (pull) code qua lại với nhau.

Mục tiêu bài học: hiểu remote repository là gì, phân biệt rõ clone, fetch, pull, push, và thực hành mô phỏng hai lập trình viên dev1, dev2 cùng làm việc với một central repo.

Mục tiêu bài học

  • Hiểu Remote Repository là gì và vì sao cần remote khi làm việc nhóm.
  • Phân biệt Working Directory, Staging Area, Local Repository và Remote Repository.
  • Sử dụng đúng các lệnh: git init --bare, git clone, git fetch, git pull, git push.
  • Nắm quy trình cộng tác cơ bản giữa nhiều lập trình viên qua một kho trung tâm.
  • Thực hành mô phỏng hai lập trình viên dev1, dev2 cùng làm việc với một central repo.

1. Remote Repository là gì?

Local Repository là kho Git nằm trên máy của bạn. Khi chạy git init hoặc git clone, máy bạn có một repository riêng, chứa lịch sử commit và cấu hình Git.

Remote Repository là kho Git nằm ở nơi khác, thường là:

  • GitHub
  • GitLab
  • Bitbucket
  • Server nội bộ công ty
  • Một thư mục bare repo dùng để mô phỏng remote khi học

Remote giúp nhiều người cùng làm việc với cùng một dự án. Mỗi người làm trên máy riêng, commit trên local, rồi đồng bộ qua remote bằng push và pull.

2. Mô hình 4 vùng trong Git

Khi học remote, nên nối kiến thức này với mô hình Git cơ bản đã học ở các bài trước:

Vùng Ý nghĩa Lệnh liên quan
Working Directory Nơi bạn sửa file trực tiếp tạo file, sửa file, xóa file
Staging Area / Index Vùng tạm để chọn file sẽ đưa vào commit git add ., git add <file>
Local Repository Lịch sử commit trên máy bạn git commit, git log
Remote Repository Kho trung tâm để chia sẻ code git clone, git pull, git push
Chiều ngược lại cũng cần nhớ: git pull hoặc git fetch đưa dữ liệu từ Remote Repository về Local Repository và Working Directory.

3. Central Repository là gì?

Central repo là kho trung tâm để nhiều lập trình viên cùng dùng. Ví dụ mô hình hai lập trình viên cùng làm việc với một central repo:

Mỗi lập trình viên có local repo riêng:

  • dev1 làm việc, commit, rồi git push lên central repo.
  • dev2 muốn lấy thay đổi mới thì chạy git pull.
  • dev2 sửa tiếp, commit, push lên central repo.
  • dev1 lại pull về để cập nhật.

4. Các lệnh Git quan trọng trong bài

Lệnh Mục đích Ghi nhớ nhanh
git init --bare <repo.git> Tạo repository dạng bare, thường dùng làm remote/central repo Bare repo không có working directory
git clone <remote> <folder> Sao chép remote repo về máy thành local repo Dùng khi bắt đầu tham gia dự án
git status Kiểm tra trạng thái file, branch, commit chưa push/pull Lệnh nên dùng thường xuyên
git add . Đưa toàn bộ thay đổi vào staging area Chuẩn bị commit
git commit -m "message" Lưu thay đổi vào local repo Commit chỉ nằm local nếu chưa push
git push Đẩy commit từ local lên remote Chia sẻ thay đổi cho người khác
git pull Kéo commit mới từ remote về local và cập nhật working directory Dùng trước khi làm việc hoặc trước khi push
git fetch Lấy thông tin commit mới từ remote nhưng chưa tự merge An toàn hơn pull khi muốn kiểm tra trước
git log --oneline --graph --all Xem lịch sử commit dạng ngắn, có nhánh Rất hữu ích khi học remote/branch

5. Phân biệt clone, fetch, pull, push

5.1. git clone

Dùng khi bạn chưa có dự án trên máy và muốn lấy toàn bộ repository từ remote.

git clone <remote-url>

Hoặc đặt tên thư mục đích:

git clone <remote-url> dev1

Sau khi clone, Git tự tạo liên kết remote mặc định tên là origin. Kiểm tra remote:

git remote -v

5.2. git fetch

Dùng để lấy thông tin mới từ remote nhưng chưa trộn vào branch hiện tại.

git fetch

Sau fetch, bạn có thể xem remote branch:

git log --oneline --graph --all

fetch phù hợp khi bạn muốn kiểm tra trước khi cập nhật code.

5.3. git pull

git pull về bản chất thường là:

git fetch + git merge

Nó vừa lấy commit mới từ remote, vừa cập nhật branch hiện tại.

git pull

5.4. git push

Dùng để đẩy commit từ local repository lên remote repository.

git push

Nếu branch local chưa có upstream, thường cần dùng:

git push -u origin main

Sau lần đầu, các lần sau có thể dùng ngắn gọn:

git push
Lưu ý: clone dùng khi chưa có repo local; pull dùng để cập nhật code mới về; push dùng để chia sẻ commit local lên remote; fetch dùng để kiểm tra remote có gì mới trước khi nhập vào code hiện tại.

6. Thực hành: mô phỏng dev1 và dev2

Phần này mô phỏng một kho trung tâm central.git và hai lập trình viên dev1, dev2 trên cùng máy. Bạn có thể làm trong PowerShell, Git Bash, hoặc terminal của VS Code.

Bước 1: Tạo thư mục bài thực hành

mkdir git-remote-practice
cd git-remote-practice

Bước 2: Tạo central repository dạng bare

git init --bare central.git

Giải thích:

  • central.git đóng vai trò remote repository.
  • --bare nghĩa là repository này không có working directory.
  • Vì không có working directory nên bạn sẽ không thấy file dự án trực tiếp như README.md, intro.ts ở thư mục này.
  • Dữ liệu commit vẫn được lưu trong cấu trúc nội bộ Git, ví dụ objects, refs, HEAD.

Bước 3: Clone central repo thành local repo của dev1 và dev2

git clone central.git dev1
git clone central.git dev2

Cấu hình thông tin riêng cho từng người:

cd dev1
git config --local user.name "Dev 1"
git config --local user.email "dev1@example.com"
cd ..

cd dev2
git config --local user.name "Dev 2"
git config --local user.email "dev2@example.com"
cd ..

Kiểm tra cấu hình:

git config --local --list

Bước 4: Dev1 tạo file đầu tiên và push lên central repo

cd dev1
echo "Day la repo cua Dev 1" > intro.ts
git status
git add .
git commit -m "Add intro file"
git push origin main

Nếu Git của bạn đang dùng branch master thay vì main, dùng:

git push origin master

Kiểm tra branch hiện tại:

git branch
Sau khi push, commit của dev1 đã nằm trên central repo.

Bước 5: Dev2 kéo thay đổi từ central repo về

cd ../dev2
git pull
ls

Kết quả mong đợi: trong dev2 xuất hiện file intro.ts do dev1 tạo.

Bước 6: Dev1 thêm file README và push

cd ../dev1
echo "Xin chao" > README.md
git add .
git commit -m "Add README"
git push

Bước 7: Dev2 pull để cập nhật README

cd ../dev2
git pull
cat README.md

Bước 8: Dev2 sửa README, thêm file mới và push

echo "Day la mot dong moi" >> README.md
echo "Vi du file moi" > example.txt
git add .
git commit -m "Update README and add example file"
git push

Bước 9: Dev1 pull để nhận thay đổi từ dev2

cd ../dev1
git status
git pull
ls
cat README.md
Sau bước này, dev1 sẽ thấy README.md có thêm dòng mới từ dev2, và có thêm file example.txt.
Vì sao cần central repo? Nếu máy dev1 hoặc dev2 hỏng, hoặc một người nghỉ việc, người mới vẫn có thể git clone từ central repo để lấy toàn bộ lịch sử dự án.

7. Làm việc với GitHub thật

Khi dùng GitHub, remote repository không phải thư mục central.git trên máy nữa mà là một URL GitHub.

Ví dụ URL HTTPS:

https://github.com/username/project-name.git

Ví dụ URL SSH:

git@github.com:username/project-name.git

7.1. Trường hợp 1: Clone repo có sẵn từ GitHub

git clone https://github.com/username/project-name.git
cd project-name

Sau đó làm việc bình thường:

git add .
git commit -m "Update project"
git push

7.2. Trường hợp 2: Đã có project local, muốn đẩy lên GitHub

git init
git add .
git commit -m "Initial commit"
git branch -M main
git remote add origin https://github.com/username/project-name.git
git push -u origin main

Kiểm tra remote:

git remote -v

8. Quy trình làm việc nhóm nên nhớ

Một workflow cơ bản khi làm chung dự án:

# 1. Trước khi sửa code, kéo bản mới nhất
git pull

# 2. Sửa code
# edit files...

# 3. Kiểm tra trạng thái
git status

# 4. Đưa file vào staging
git add .

# 5. Commit
git commit -m "Meaningful commit message"

# 6. Đẩy lên remote
git push

Quy tắc thực tế:

  1. Luôn git pull trước khi bắt đầu làm nếu bạn làm trên branch chung.
  2. Commit nhỏ, rõ nghĩa, mỗi commit nên giải quyết một việc.
  3. Không nên sửa trực tiếp trên main trong dự án thật; nên tạo branch riêng.
  4. Không dùng git push --force nếu chưa hiểu rõ hậu quả.
  5. Trước khi push, chạy test/build nếu dự án có test/build.

9. Commit message nên viết thế nào?

Commit message nên ngắn, rõ hành động.

Ví dụ tốt:

git commit -m "Add login form"
git commit -m "Fix routing table import bug"
git commit -m "Update README with setup guide"
git commit -m "Refactor Git remote practice notes"

Ví dụ chưa tốt:

git commit -m "01"
git commit -m "fix"
git commit -m "update"
Tránh các commit message mơ hồ như "fix" hay "update". Hãy mô tả rõ thay đổi đã làm, ví dụ "Add intro file", "Update README and add example file".

10. Các lỗi thường gặp khi học remote

10.1. fatal: not a git repository

Nguyên nhân: bạn đang đứng ngoài thư mục Git. Cách kiểm tra:

pwd
ls -a

Phải thấy thư mục .git nếu là repo thường. Với bare repo thì thấy các thư mục như objects, refs, hooks ngay ở gốc.

10.2. remote origin already exists

Nguyên nhân: repo đã có remote tên origin. Kiểm tra:

git remote -v

Đổi URL remote:

git remote set-url origin <new-url>

Hoặc xóa rồi thêm lại:

git remote remove origin
git remote add origin <url>

10.3. src refspec main does not match any

Nguyên nhân thường gặp: chưa có commit nào, hoặc branch hiện tại không tên là main. Cách xử lý:

git status
git branch
git add .
git commit -m "Initial commit"
git branch -M main
git push -u origin main

10.4. failed to push some refs

Nguyên nhân phổ biến: remote có commit mới mà local chưa có. Cách xử lý an toàn:

git pull
# xử lý conflict nếu có
git push

10.5. Conflict khi git pull

Nếu hai người sửa cùng một vùng trong cùng một file, Git có thể báo conflict. File conflict có dạng:

<<<<<<< HEAD
Nội dung của bạn
=======
Nội dung từ remote
>>>>>>> origin/main

Cách xử lý: mở file, chọn nội dung đúng, xóa các dòng đánh dấu conflict, rồi chạy:

git add .
git commit -m "Resolve merge conflict"
git push
Bài sau sẽ đi sâu hơn vào cách xử lý merge conflict một cách chi tiết.

11. Bảng so sánh nhanh

Tình huống Lệnh nên dùng
Muốn tải một repo về máy lần đầu git clone <url>
Muốn xem local đang thay đổi gì git status
Muốn lưu thay đổi vào lịch sử local git add . → git commit -m "..."
Muốn đẩy commit lên GitHub/remote git push
Muốn kéo code mới từ remote về git pull
Muốn kiểm tra remote có gì mới nhưng chưa nhập vào code git fetch
Muốn xem remote đang liên kết git remote -v
Muốn đổi URL remote git remote set-url origin <url>

12. Bài tập tự luyện

Bài 1: Mô phỏng lại video

Tạo một thư mục mới và thực hiện lại toàn bộ quy trình:

  1. Tạo central.git bằng git init --bare.
  2. Clone thành dev1, dev2.
  3. Dev1 tạo file, commit, push.
  4. Dev2 pull về.
  5. Dev2 sửa file, commit, push.
  6. Dev1 pull về.

Yêu cầu cuối, bạn phải nhìn thấy lịch sử commit đã được đồng bộ giữa dev1, dev2:

git log --oneline --graph --all

Bài 2: Phân biệt fetch và pull

Trong dev1, tạo thêm một commit và push. Sang dev2, chạy:

git fetch
git status
git log --oneline --graph --all

Sau đó mới chạy:

git pull

Ghi lại khác biệt trước và sau khi pull.

Bài 3: Tạo lỗi để hiểu lỗi

Cố tình để dev1 và dev2 cùng sửa một dòng trong README.md, sau đó push từ một bên trước. Bên còn lại chạy git pull để quan sát conflict. Mục tiêu: biết đọc ký hiệu conflict và tự xử lý.

Tổng kết

Sau bài này, bạn cần nắm chắc:

  • Remote Repository là kho trung tâm để nhiều người cùng làm việc, khác với Local Repository nằm trên máy bạn.
  • git clone dùng khi chưa có repo local.
  • git pull dùng để cập nhật code mới từ remote về (tương đương fetch + merge).
  • git push dùng để chia sẻ commit local lên remote.
  • git fetch dùng để kiểm tra remote có gì mới trước khi nhập vào code hiện tại.
  • Commit chỉ lưu ở local; muốn người khác thấy thì phải push.
  • Remote mặc định sau khi clone thường có tên là origin.
Quy trình chuẩn khi làm việc nhóm: git pull → sửa code → git status → git add . → git commit -m "message" → git push.
