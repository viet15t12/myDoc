7. Xử lý xung đột trong Git - Merge Conflict

Merge Conflict là gì, vì sao xảy ra, cách đọc ký hiệu <<<<<<<, =======, >>>>>>>, và cách xử lý xung đột giữa hai lập trình viên dùng chung một Central Repo.

Giới thiệu

Ở bài trước, bạn đã thực hành pull, push giữa nhiều người dùng chung một Central Repo. Khi mọi người làm việc theo trình tự rõ ràng, Git thường tự ghép thay đổi rất êm. Nhưng trong thực tế, hai người có thể cùng sửa một vùng nội dung trên cùng một phiên bản cũ — đó là lúc Merge Conflict xuất hiện.

Conflict không phải lỗi hỏng Git. Đó là lúc Git không tự quyết định được nên giữ nội dung nào, và cần con người quyết định thay.

Mục tiêu bài học

  • Hiểu Merge Conflict là gì và vì sao xảy ra khi nhiều người cùng sửa một file.
  • Mô phỏng được tình huống xung đột với một Central Repo và hai developer dev1, dev2.
  • Đọc đúng ký hiệu xung đột: <<<<<<< HEAD, =======, >>>>>>> branch-name.
  • Biết cách sửa conflict thủ công, sau đó git add, git commit, git push.
  • Biết cách hạn chế xung đột khi làm việc nhóm.

2. Merge Conflict là gì?

Khi nhiều lập trình viên cùng cộng tác, mỗi người thường xuyên pull, sửa file, commit, rồi push. Nếu Dev1 sửa xong và đẩy lên trước, Dev2 pull về rồi mới sửa tiếp, Git thường không gặp vấn đề.

Nhưng nếu hai người cùng lấy một phiên bản cũ của file về máy, rồi mỗi người sửa cùng một vùng nội dung, người push trước sẽ thành công. Người push sau sẽ bị từ chối vì lệch phiên bản so với remote, và khi pull về để cập nhật, Git có thể không tự ghép được — đó là Merge Conflict.

Sơ đồ tình huống: một Central Repo, hai Developer

Dev1

local repo

⇄

Central Repo

a.txt

⇄

Dev2

local repo

Cả Dev1 và Dev2 cùng clone/pull file a.txt về từ Central Repo. Nếu cả hai cùng sửa file này dựa trên cùng một phiên bản cũ, nguy cơ conflict xuất hiện ngay khi người thứ hai cố push hoặc pull.

3. Mô phỏng trường hợp chưa có xung đột

Luồng làm việc bình thường, chưa gây conflict:

  1. Dev1 tạo file a.txt, ghi nội dung ban đầu Hello.
  2. Dev1 add, commit, rồi push lên Central Repo.
  3. Dev2 pull về.
  4. Dev2 thấy file a.txt đã được cập nhật, chưa có gì xung đột.
Vì Dev2 đã lấy phiên bản mới nhất trước khi sửa, nên Git không có gì phải ghép xung đột.

4. Cố tình tạo Merge Conflict

Tình huống thực tế hơn — hai người cùng sửa file dựa trên bản cũ:

  1. Dev1 sửa file a.txt, thêm dòng world, rồi commit và push lên Central Repo.
  2. Dev2 lúc này chưa pull phiên bản mới về.
  3. Dev2 vẫn sửa file cũ của mình, thêm dòng Lê Nhật Tùng, rồi commit thay đổi local.
  4. Dev2 thử git push nhưng bị từ chối vì remote đã có commit mới từ Dev1.
  5. Dev2 chạy git pull, Git cố tự trộn file nhưng không quyết định được nội dung nào nên báo conflict.

Lỗi thường gặp khi push bị từ chối:

! [rejected]        main -> main (fetch first)
error: failed to push some refs
hint: Updates were rejected because the remote contains work that you do not have locally.

Khi chạy git pull, Git có thể báo:

Auto-merging a.txt
CONFLICT (content): Merge conflict in a.txt
Automatic merge failed; fix conflicts and then commit the result.
Đây chính là thời điểm xảy ra conflict: remote đang giữ Hello + world, còn local của Dev2 đang giữ Hello + Lê Nhật Tùng. Git không biết bạn muốn giữ dòng nào.

5. Cách đọc ký hiệu xung đột (conflict marker)

Sau khi Git báo conflict, mở file bị lỗi bằng Notepad hoặc VS Code. Bên trong file sẽ có các ký hiệu đặc biệt do Git chèn vào để đánh dấu vùng xung đột. 

Hello
<<<<<<< HEAD
Lê Nhật Tùng
=======
world
>>>>>>> origin/main
VùngÝ nghĩa
<<<<<<< HEADBắt đầu vùng xung đột, đánh dấu nội dung local hiện tại.
Nội dung ngay dưới HEADNội dung ở nhánh/local hiện tại của bạn.
=======Đường phân cách giữa hai phiên bản.
Nội dung ngay dưới dấu phân cáchNội dung đến từ remote (origin/main).
>>>>>>> origin/mainKết thúc vùng xung đột, đánh dấu nguồn của phần remote.

Dòng không nằm trong vùng conflict, ví dụ Hello, là dòng không bị xung đột.

Người lập trình viên phải tự quyết định:

  • Giữ nội dung local.
  • Giữ nội dung remote.
  • Giữ cả hai.
  • Viết lại thành nội dung mới đúng hơn.

Ví dụ sau khi xử lý, giữ lại cả ba dòng và xóa hết marker:

Hello
world
Lê Nhật Tùng

Sau đó lưu file, chạy:

git add a.txt
git commit -m "resolve merge conflict in a.txt"
git push
Khi đó conflict đã được xử lý và lịch sử Git tiếp tục làm việc bình thường.

6. Quy trình chuẩn 5 bước khi gặp conflict

Bước 1 — Kiểm tra trạng thái

git status

Git sẽ cho biết file nào đang bị conflict, ví dụ:

both modified:   a.txt

Bước 2 — Mở file bị conflict

Mở bằng VS Code:

code a.txt

Hoặc mở cả thư mục:

code .

Trên Windows cũng có thể mở bằng Notepad:

notepad a.txt

Bước 3 — Đọc marker conflict

<<<<<<< HEAD
Nội dung của bạn
=======
Nội dung từ nhánh/remote được merge vào
>>>>>>> origin/main

Bước 4 — Sửa file về trạng thái cuối cùng

Tuyệt đối không để lại các dòng marker này trong file:

<<<<<<< HEAD
=======
>>>>>>> origin/main

Bước 5 — Đánh dấu đã xử lý và commit

git add a.txt
git commit -m "resolve merge conflict in a.txt"
git push

Nếu conflict xảy ra khi đang merge hoặc pull, commit sau khi sửa conflict chính là commit hoàn tất quá trình merge.

7. Thực hành đầy đủ trên Windows PowerShell

Mô phỏng lại đúng tinh thần bài học: một repo trung tâm và hai thư mục local đại diện cho dev1, dev2.

Bước 1: Tạo thư mục bài tập

mkdir bai-tap-06
cd bai-tap-06

Bước 2: Tạo Central Repository dạng bare repo

git init --bare central-repo.git

Bare repo chỉ dùng để nhận push và cung cấp dữ liệu cho pull/clone, không dùng để sửa file trực tiếp.

Bước 3: Clone thành hai thư mục Dev1 và Dev2

git clone .\central-repo.git dev1
git clone .\central-repo.git dev2

Cấu trúc thư mục sau khi clone:

bai-tap-06/
├── central-repo.git/
├── dev1/
└── dev2/

Bước 4: Dev1 tạo file và push lần đầu

cd dev1

"Hello" > a.txt

git add a.txt
git commit -m "dev1 add a.txt"

git branch -M main
git push -u origin main

Bước 5: Dev2 pull về để có file mới

cd ..\dev2

git pull origin main
type a.txt

Kết quả mong muốn:

Hello

Bước 6: Dev1 sửa file và push trước

cd ..\dev1

"world" >> a.txt

git add a.txt
git commit -m "dev1 add world"
git push

Lúc này remote đã có nội dung:

Hello
world

Bước 7: Dev2 chưa pull nhưng vẫn sửa file cũ

cd ..\dev2

"Lê Nhật Tùng" >> a.txt

git add a.txt
git commit -m "dev2 edit a.txt"

Dev2 đang có nội dung local Hello / Lê Nhật Tùng, còn remote đang có Hello / world.

Bước 8: Dev2 push và bị từ chối

git push
! [rejected] main -> main (fetch first)
error: failed to push some refs

Nguyên nhân: remote đã có commit mới từ Dev1 mà Dev2 chưa có.

Bước 9: Dev2 pull về và gặp conflict

git pull origin main
Auto-merging a.txt
CONFLICT (content): Merge conflict in a.txt
Automatic merge failed; fix conflicts and then commit the result.

Bước 10: Mở file và sửa conflict

notepad a.txt

File đang như sau:

Hello
<<<<<<< HEAD
Lê Nhật Tùng
=======
world
>>>>>>> origin/main

Sửa thành (xóa hết marker, giữ cả hai dòng):

Hello
world
Lê Nhật Tùng

Lưu file lại.

Bước 11: Hoàn tất xử lý conflict

git add a.txt
git commit -m "resolve merge conflict in a.txt"
git push

Kiểm tra lại:

git status
git log --oneline --graph --all
Conflict đã được xử lý, lịch sử Dev1/Dev2 đồng bộ trở lại trên Central Repo.

8. Khi nào dễ và khi nào khó bị conflict

Git thường merge tự động được nếu...Git dễ bị conflict nếu...
Hai người sửa hai file khác nhau. Hai người sửa cùng một dòng.
Hai người sửa hai vùng khác nhau trong cùng một file. Một người sửa file, người kia xóa file.
Một người thêm dòng mới, người kia sửa vùng không liên quan. Hai nhánh cùng đổi tên hoặc di chuyển file theo cách khác nhau.
— Một người sửa cấu trúc lớn, người kia cũng sửa dựa trên cấu trúc cũ.
Nói đơn giản: conflict không phải lỗi hỏng Git, mà là Git yêu cầu con người quyết định phiên bản cuối cùng.

9. Cách hạn chế Merge Conflict khi làm việc nhóm

Cách làmTác dụng
git pull trước khi bắt đầu sửaGiúp local mới nhất, giảm nguy cơ sửa trên nền cũ.
Commit nhỏ, thường xuyênDễ merge, dễ review, dễ rollback.
Không để một file quá nhiều người sửa cùng lúcGiảm khả năng sửa cùng dòng.
Chia task rõ ràngMỗi người phụ trách vùng code khác nhau.
Thường xuyên trao đổi khi sửa file quan trọngTránh hai người cùng sửa cùng logic.
Dùng branch riêng cho từng tính năngGiúp cô lập thay đổi.
Dùng Pull Request / Merge RequestCó bước review trước khi merge vào nhánh chính.
Format code thống nhấtTránh conflict do khác format.
Không sửa file sinh tự động nếu không cầnFile lock, file build, file generated dễ gây conflict.

10. Một số lệnh hữu ích khi đang bị conflict

Xem file nào bị conflict

git status

Xem chi tiết khác biệt

git diff

Mở VS Code để sửa conflict

code .

VS Code thường hiện các nút:

  • Accept Current Change
  • Accept Incoming Change
  • Accept Both Changes
  • Compare Changes

Hủy quá trình merge nếu chưa muốn xử lý

git merge --abort
Chỉ dùng git merge --abort khi bạn muốn quay về trạng thái trước khi merge.

Kiểm tra lịch sử sau khi xử lý

git log --oneline --graph --all

Đọc nhanh lỗi khi push bị từ chối

Updates were rejected because the remote contains work that you do not have locally.

Hiểu là: remote đã có commit mới, local của bạn đang cũ hơn remote, và bạn cần pull trước khi push được tiếp.

git pull
# sửa conflict nếu có
git add .
git commit -m "resolve merge conflict"
git push

11. Lưu ý quan trọng khi xử lý conflict

11.1. Không xóa bừa toàn bộ nội dung

Khi thấy marker conflict, không nên xóa hết một bên nếu chưa hiểu ý nghĩa code. Cần đọc kỹ: phần nào của mình, phần nào của người khác, có cần giữ cả hai không, và giữ cả hai có làm sai logic chương trình không.

11.2. Không commit khi marker conflict vẫn còn

Trước khi commit, nên tìm trong toàn bộ project:

git grep "<<<<<<<"
git grep "======="
git grep ">>>>>>>"

Nếu còn marker, nghĩa là conflict chưa xử lý sạch.

11.3. Sau khi sửa conflict nên chạy/test lại chương trình

Git chỉ biết trộn text, không biết chương trình của bạn còn đúng logic hay không. Sau khi sửa conflict nên chạy lại chương trình, chạy test nếu có, và kiểm tra file có bị mất dòng quan trọng hoặc sai định dạng không.

11.4. Commit message nên rõ ràng

git commit -m "resolve merge conflict in a.txt"
Không nên dùng message quá mơ hồ như git commit -m "fix".

Tổng kết

Merge Conflict xảy ra khi Git không tự quyết định được cách ghép thay đổi. Trong bài này, conflict được mô phỏng bằng một Central Repo và hai developer cùng sửa file a.txt: Dev1 push trước, Dev2 sửa trên bản cũ rồi push bị từ chối; khi Dev2 pull về, Git báo conflict trong file.

  • Marker conflict gồm <<<<<<< HEAD (local), ======= (phân cách), >>>>>>> origin/main (remote).
  • Phải xóa hết marker trước khi commit, và chỉ giữ nội dung đúng cuối cùng.
  • Quy trình chuẩn: git status → mở file → đọc marker → sửa file → git add → git commit → git push.
  • Có thể hạn chế conflict bằng cách pull thường xuyên, commit nhỏ, chia task rõ ràng, dùng branch riêng và Pull Request.
Điểm quan trọng nhất: Git chỉ phát hiện và đánh dấu vùng xung đột; chính lập trình viên phải đọc nội dung và quyết định giữ phần nào, bỏ phần nào, hoặc kết hợp cả hai.
