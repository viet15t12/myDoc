10. Git Merge - Kết hợp nội dung từ các nhánh

git merge dùng để gộp nội dung từ branch khác vào branch hiện tại — gồm fast-forward merge, three-way merge, và cách xử lý conflict khi cả hai nhánh cùng sửa một vùng nội dung.

Giới thiệu

Bài trước đã học branch — cách tách ra nhiều hướng phát triển song song. Bài này tập trung vào cách kết hợp các branch lại với nhau bằng lệnh git merge, kể cả khi hai nhánh cùng sửa một vùng nội dung và Git báo conflict.

Mục tiêu bài học

  • git merge dùng để gộp nội dung từ branch khác vào branch hiện tại.
  • Muốn merge branch B vào branch A thì phải checkout sang A trước, sau đó chạy git merge B.
  • Git có thể tự merge nếu thay đổi không đụng nhau.
  • Nếu thay đổi đụng cùng file/cùng dòng, Git báo conflict và cần sửa thủ công.
  • Sau khi sửa conflict, phải git add và git commit để hoàn tất merge.
  • Cách xem lịch sử sau merge bằng git log --oneline --graph --all.

1. git merge là gì?

git merge là lệnh dùng để đưa thay đổi từ một branch vào branch hiện tại.

Cú pháp:

git merge <ten-nhanh-muon-gop-vao>

Ví dụ đang ở master, muốn gộp nhanh-b vào master:

git checkout master
git merge nhanh-b
Nói ngắn gọn: đang đứng ở branch nào thì branch đó là nơi nhận thay đổi.

2. Từ khóa quan trọng

Từ khóaÝ nghĩa
mergeGộp nội dung/lịch sử từ branch khác vào branch hiện tại.
current branchBranch bạn đang đứng. Đây là branch nhận thay đổi.
source branchBranch được đưa vào branch hiện tại.
merge commitCommit đặc biệt sinh ra khi Git gộp hai nhánh theo kiểu 3-way merge.
fast-forwardTrường hợp merge đơn giản, Git chỉ kéo con trỏ branch tiến lên.
conflictXung đột khi Git không tự quyết định được nội dung cuối cùng.
git statusKiểm tra trạng thái, đặc biệt hữu ích khi merge conflict.
git log --graph --allXem lịch sử nhánh sau merge.

3. Mô hình merge A/B

Có thể hiểu mô hình merge như sau:

  • Nhánh A là nhánh chính, ví dụ master. Ban đầu có commit A1, sau đó A2.
  • Từ A2, tách ra nhánh B. Nhánh B có commit B1, B2.
  • Trong lúc đó, nhánh A vẫn tiếp tục có commit A3, A4.
  • Đến lúc cần gộp, ta merge B vào A.

master (A)

A1 → A2 → A3 → A4

⤵

B

(từ A2) → B1 → B2

git checkout master
git merge B

4. Thực hành: tạo commit A1, A2 và branch B

Bước 1: Tạo repo và commit A1, A2

mkdir merge-practice
cd merge-practice

git init --bare central-repo.git

git clone central-repo.git dev1
cd dev1

Tạo commit A1:

echo "A1" > main.txt

git add .
git commit -m "A1"
git push

Tạo commit A2:

echo "A2" >> main.txt

git add .
git commit -m "A2"
git push

Xem lịch sử:

git log --oneline

Bước 2: Tạo nhánh B và tạo commit B1, rồi quay lại master tạo A3/A4

git branch B
git checkout B

Tạo commit B1:

echo "B1" >> main.txt

git add .
git commit -m "B1"

Nếu cần push branch B lần đầu:

git push -u origin B

Quay lại nhánh chính, tạo commit A3 và A4:

git checkout master

echo "A3" >> main.txt
git add .
git commit -m "A3"

echo "A4" >> main.txt
git add .
git commit -m "A4"

Bước 3: Quay lại nhánh B tạo B2

git checkout B

echo "B2" >> main.txt
git add .
git commit -m "B2"

Lúc này lịch sử có hai hướng:

  • master: A1 -> A2 -> A3 -> A4
  • B: A1 -> A2 -> B1 -> B2

Xem bằng:

git log --oneline --graph --all

5. Merge nhánh B vào master và xử lý conflict

Muốn đưa công việc từ B vào master, trước hết phải đứng ở master:

git checkout master

Sau đó merge B vào master:

git merge B

Nếu Git tự merge được, quá trình hoàn tất nhanh. Nhưng vì cả hai nhánh cùng sửa file main.txt, Git báo conflict:

Auto-merging main.txt
CONFLICT (content): Merge conflict in main.txt
Automatic merge failed; fix conflicts and then commit the result.

Mở file main.txt, có thể thấy dạng:

A1
A2
<<<<<<< HEAD
A3
A4
=======
B1
B2
>>>>>>> B

Phần HEAD là nội dung của branch hiện tại (master); phần dưới ======= là nội dung đến từ branch B. Người dùng phải sửa lại nội dung cuối cùng, ví dụ: 

A1
A2
B1
A3
A4
B2

(hoặc thứ tự khác tùy logic dự án). Sau đó hoàn tất merge:

git add main.txt
git commit -m "resolve merge conflict and merge B into master"

6. Cách hiểu thứ tự commit sau merge

Sau khi merge, git log có thể hiển thị các commit không hoàn toàn theo "nhánh A trước rồi nhánh B sau". Git sắp xếp theo quan hệ cha-con và thời gian commit.

  • Merge không đơn giản là copy file.
  • Merge kết hợp lịch sử của hai branch.
  • Nếu có conflict, bạn quyết định nội dung cuối cùng.
  • Sau khi commit merge, branch nhận merge chứa cả lịch sử của branch được merge.

Xem lịch sử rõ nhất bằng:

git log --oneline --graph --all --decorate

7. Hai kiểu merge phổ biến

7.1. Fast-forward merge

Xảy ra khi branch hiện tại chưa có commit mới nào kể từ lúc tách branch.

main:    A1---A2
feature:       A3---A4

Khi merge:

git checkout main
git merge feature

Git chỉ kéo main tiến lên A4, không cần tạo merge commit.

7.2. Three-way merge

Xảy ra khi cả hai branch đều có commit mới sau khi tách nhánh.

main:    A1---A2---A3---A4
feature:      \---B1---B2

Khi merge, Git tạo merge commit để nối hai lịch sử lại với nhau.

8. Quy trình chuẩn khi merge branch

Bước 1 — Kiểm tra trạng thái

git status

Chỉ nên merge khi working tree sạch.

Bước 2 — Cập nhật branch nhận merge

git checkout master
git pull
# hoặc
git checkout main
git pull

Bước 3 — Merge branch khác vào

git merge ten-nhanh

Ví dụ:

git merge B

Bước 4 — Nếu có conflict, sửa file

git status

Mở file và xóa marker, giữ nội dung đúng:

<<<<<<< HEAD
...
=======
...
>>>>>>> B

Bước 5 — Hoàn tất merge

git add .
git commit -m "merge B into master"

Nếu Git đã tự tạo merge commit message, chỉ cần lưu và đóng editor.

Bước 6 — Push lên remote

git push

9. Hủy merge nếu chưa muốn xử lý

Nếu merge bị conflict và bạn muốn quay lại trạng thái trước khi merge:

git merge --abort
Chỉ dùng git merge --abort khi bạn chưa muốn xử lý conflict lúc này.

10. Khi nào nên merge?

Nên merge khi:

  • Tính năng trên branch phụ đã hoàn thành.
  • Code đã test ổn.
  • Branch phụ đã cập nhật với branch chính.
  • Không còn thay đổi đang dở dang trong working tree.
Trong làm việc nhóm, thường không merge trực tiếp bằng terminal vào main/master, mà sẽ mở Pull Request trên GitHub/GitLab để review trước.

11. Merge Conflict trong bài này liên hệ với bài Merge Conflict trước

Bài trước đã học cách xử lý conflict khi git pull gặp xung đột. Bài này cho thấy conflict cũng có thể xảy ra khi git merge branch. Bản chất giống nhau:

  • Git đang cố gộp hai nguồn thay đổi.
  • Có vùng nội dung Git không tự quyết định được.
  • Git chèn marker conflict vào file.
  • Người dùng sửa tay.
  • Sau đó git add và git commit.

Công thức nhớ:

git status
# sửa file conflict
git add <file>
git commit

12. Bài tập tự luyện

Bài 1: Merge không conflict

mkdir merge-no-conflict
cd merge-no-conflict

git init

echo "home" > index.html
git add .
git commit -m "initial"

git checkout -b feature-css
echo "body { font-family: Arial; }" > style.css
git add .
git commit -m "add css"

git checkout master
git merge feature-css

Quan sát: Git có thể merge tự động vì hai branch không sửa cùng file.

Bài 2: Merge có conflict

mkdir merge-conflict-demo
cd merge-conflict-demo

git init

echo "Title: Git Course" > README.md
git add .
git commit -m "initial readme"

git checkout -b branch-a
echo "Title: Git Branch Course" > README.md
git add .
git commit -m "edit title in branch a"

git checkout master
echo "Title: Git Merge Course" > README.md
git add .
git commit -m "edit title in master"

git merge branch-a

Sau đó mở README.md, sửa conflict, rồi:

git add README.md
git commit -m "resolve title conflict"

Bài 3: Xem graph sau merge

Sau mỗi lần merge, chạy:

git log --oneline --graph --all --decorate

Ghi lại sự khác nhau giữa: merge không conflict, merge có conflict, fast-forward merge, và three-way merge.

Tổng kết

git merge dùng để gộp nội dung từ branch khác vào branch hiện tại. Muốn merge B vào master thì phải đứng ở master rồi chạy git merge B.

  • Nếu Git tự gộp được (fast-forward hoặc không đụng nội dung), quá trình rất nhanh.
  • Nếu hai branch sửa cùng vùng nội dung, Git báo conflict và tạo merge commit sau khi xử lý (three-way merge).
  • Khi conflict: mở file, xóa marker, giữ nội dung đúng, rồi git add và git commit.
  • Có thể hủy merge dở dang bằng git merge --abort.
Trong làm việc nhóm thật, nên dùng Pull Request để review trước khi merge vào nhánh chính, thay vì merge trực tiếp bằng terminal.
