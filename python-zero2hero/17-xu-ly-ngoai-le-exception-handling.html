17. Xử lý Ngoại lệ (Exception Handling) trong Python

Exception handling — cơ chế xử lý lỗi runtime trong Python: try, except, else, finally, raise, và custom exception. Giúp chương trình không bị crash khi gặp lỗi, cung cấp thông báo thân thiện cho người dùng.

Giới thiệu

Ở bài trước, chúng ta đã học cách nhận dạng 10 lỗi phổ biến trong Python. Bài này đi tiếp: làm thế nào để xử lý những lỗi đó một cách có kiểm soát — thay vì để chương trình crash?

Trong Python, có hai loại lỗi cần phân biệt:

Loại lỗiKhi nào xảy raVí dụCó xử lý được không?
Syntax Error Trước khi chạy (lúc parse) Thiếu :, thiếu () Không — phải sửa code
Exception (Runtime Error) Trong khi chạy ZeroDivisionError, KeyError… Có — dùng try/except

Exception handling (xử lý ngoại lệ) là cơ chế để bắt và xử lý các exception xảy ra khi chạy, giúp chương trình tiếp tục hoạt động thay vì bị crash.

1. Tại sao cần xử lý ngoại lệ?

Không xử lý → chương trình crash

# Không có exception handling
so = int(input("Nhập một số: "))   # Người dùng nhập "abc"
ket_qua = 100 / so
print(f"Kết quả: {ket_qua}")

# Khi nhập "abc":
# Traceback (most recent call last):
#   ...
# ValueError: invalid literal for int() with base 10: 'abc'
# → Chương trình dừng đột ngột!

Có xử lý → chương trình phản hồi thân thiện

# Có exception handling
try:
    so = int(input("Nhập một số: "))
    ket_qua = 100 / so
    print(f"Kết quả: {ket_qua}")
except ValueError:
    print("Bạn cần nhập một số nguyên!")
except ZeroDivisionError:
    print("Không thể chia cho 0!")
# → Chương trình vẫn chạy bình thường, thông báo thân thiện
3 lý do chính cần exception handling:
  • Tránh crash — chương trình không bị dừng đột ngột khi gặp lỗi runtime.
  • UX tốt hơn — người dùng nhận được thông báo hiểu được, không phải Traceback kỹ thuật.
  • Debug dễ hơn — bạn kiểm soát được loại lỗi nào xảy ra và ghi log để trace.

2. Cấu trúc đầy đủ

Python cung cấp 4 keyword cho exception handling:

try:
    # Code có thể gây ra lỗi
    pass

except LoaiLoi:
    # Chạy khi try gặp lỗi khớp với LoaiLoi
    pass

else:
    # Chạy khi try KHÔNG có lỗi
    pass

finally:
    # LUÔN chạy, dù có lỗi hay không
    pass
KeywordBắt buộc?Khi nào chạyDùng để
try Có Luôn chạy đầu tiên Đặt code có thể gây lỗi vào đây
except Có Chỉ khi try gặp lỗi Xử lý lỗi, thông báo cho người dùng
else Không Chỉ khi try KHÔNG có lỗi Code chạy khi mọi thứ ổn
finally Không Luôn luôn — dù có lỗi hay không Dọn dẹp: đóng file, đóng kết nối DB…
Lưu ý: try bắt buộc phải đi kèm ít nhất một except hoặc một finally. Chỉ có try một mình sẽ báo SyntaxError.

3. try / except — Cơ bản

Cách 1: except không chỉ định lỗi cụ thể

my_dict = {"age": 10, "name": "Minh"}

try:
    my_value = my_dict["address"]   # Key không tồn tại → KeyError
except:
    print("Đi vào đây khi xảy ra bất kỳ lỗi gì")

print("Bye bye")   # Vẫn chạy bình thường

Khi không chỉ định loại lỗi, except bắt mọi loại lỗi xảy ra trong try.

Cách 2: except chỉ định lỗi cụ thể (khuyến nghị)

my_dict = {"age": 10, "name": "Minh"}

try:
    my_value = my_dict["address"]   # → KeyError
except ZeroDivisionError:
    print("Lỗi chia cho 0")
    # Không khớp với KeyError → không vào đây!

# → Chương trình vẫn crash vì KeyError không được xử lý
Quan trọng: Nếu except chỉ định sai loại lỗi, lỗi vẫn sẽ propagate và crash chương trình. Phải chỉ định đúng loại lỗi cần xử lý. 
# Nên - chỉ định đúng loại lỗi
try:
    my_value = my_dict["address"]
except KeyError:
    print("Lỗi KeyError: key không tồn tại trong dictionary")
Best practice: Luôn chỉ định loại lỗi cụ thể trong except. Điều này giúp code rõ ràng hơn, dễ debug hơn, và không vô tình "nuốt" các lỗi ngoài ý muốn.

4. Nhiều except — Xử lý từng loại lỗi khác nhau

Một khối try có thể có nhiều except để xử lý các loại lỗi khác nhau. Python sẽ khớp từ trên xuống và chỉ chạy một except đầu tiên khớp. 

try:
    so = int(input("Nhập vào một số: "))   # Nhập "abc" → ValueError
    ket_qua = 100 / so                      # Nhập 0    → ZeroDivisionError
    print(f"Kết quả: {ket_qua}")

except ZeroDivisionError:
    print("Không thể chia cho 0!")

except ValueError:
    print("Nhập sai rồi. Hãy nhập một số thôi!")

# Test 1: Nhập 0
# → Không thể chia cho 0!

# Test 2: Nhập "abc"
# → Nhập sai rồi. Hãy nhập một số thôi!

# Test 3: Nhập 10
# → Kết quả: 10.0

Có thể bắt nhiều loại lỗi trong một except bằng cách dùng tuple:

try:
    ket_qua = int(input("Nhập số: ")) / int(input("Nhập số chia: "))
except (ValueError, ZeroDivisionError) as e:
    print(f"Lỗi nhập liệu: {e}")

5. Bắt thông tin lỗi – except … as e

Thêm as e (hoặc as err) sau tên lỗi để lưu object lỗi vào biến và lấy thông tin chi tiết: 

try:
    ket_qua = 10 / 0
except ZeroDivisionError as e:
    print(f"Loại lỗi: {type(e).__name__}")   # ZeroDivisionError
    print(f"Mô tả lỗi: {e}")                 # division by zero
    print(f"Chi tiết: {e.args}")             # ('division by zero',)

Ứng dụng thực tế — ghi log lỗi:

import logging

logging.basicConfig(filename="app.log", level=logging.ERROR)

def xu_ly_du_lieu(data):
    try:
        ket_qua = int(data) * 2
        return ket_qua
    except ValueError as e:
        logging.error(f"Lỗi chuyển đổi dữ liệu: {e} | Input: {data!r}")
        return None

print(xu_ly_du_lieu("42"))     # 84
print(xu_ly_du_lieu("abc"))   # None (và ghi vào app.log)

6. else — Chạy khi không có lỗi

Phần else chỉ chạy khi khối try hoàn thành mà không có lỗi nào xảy ra. Đây là nơi đặt code phụ thuộc vào thành công của try. 

try:
    so = int(input("Nhập vào một số: "))
    ket_qua = 100 / so

except ZeroDivisionError:
    print("Không thể chia cho 0!")

except ValueError:
    print("Nhập sai rồi. Hãy nhập một số thôi!")

else:
    # Chỉ chạy khi try thành công (không có lỗi)
    print(f"Nhập số ok! Phép chia ok! Kết quả: {ket_qua}")

# Test nhập 10:
# → Nhập số ok! Phép chia ok! Kết quả: 10.0

# Test nhập 0:
# → Không thể chia cho 0!  (else không chạy)
Tại sao có else? Có thể bạn thắc mắc: tại sao không đặt code đó vào cuối try? Vì nếu code trong else gây ra lỗi, lỗi đó sẽ không bị bắt bởi các except ở trên — giúp phân biệt rõ ràng lỗi nào đến từ đâu.

7. finally — Luôn luôn chạy

Phần finally luôn được thực thi — dù try thành công hay thất bại. Dùng để dọn dẹp tài nguyên: đóng file, đóng kết nối database, giải phóng bộ nhớ… 

try:
    so = int(input("Nhập vào một số: "))
    ket_qua = 100 / so

except ZeroDivisionError:
    print("Không thể chia cho 0!")

except ValueError:
    print("Nhập sai rồi. Hãy nhập một số thôi!")

else:
    print(f"Nhập số ok! Phép chia ok! Kết quả: {ket_qua}")

finally:
    print("Đi vào đây bất kể có lỗi hay không")

# Test nhập 10:
# → Nhập số ok! Phép chia ok! Kết quả: 10.0
# → Đi vào đây bất kể có lỗi hay không

# Test nhập 0:
# → Không thể chia cho 0!
# → Đi vào đây bất kể có lỗi hay không

# Test nhập "abc":
# → Nhập sai rồi. Hãy nhập một số thôi!
# → Đi vào đây bất kể có lỗi hay không

Ví dụ thực tế nhất của finally — đóng kết nối database:

import sqlite3

connection = None
try:
    connection = sqlite3.connect("data.db")
    cursor = connection.cursor()
    cursor.execute("SELECT * FROM users")
    rows = cursor.fetchall()

except sqlite3.Error as e:
    print(f"Lỗi database: {e}")

else:
    for row in rows:
        print(row)

finally:
    # Luôn đóng kết nối dù có lỗi hay không
    if connection:
        connection.close()
        print("Đã đóng kết nối database.")
finally vs context manager: Với file, dùng with open(...) as f thay vì try/finally vì ngắn gọn hơn. Với database hoặc kết nối mạng, finally vẫn thường được dùng.

8. raise — Tự phát sinh lỗi

Đôi khi bạn muốn chủ động báo lỗi khi dữ liệu không hợp lệ, thay vì để lỗi âm thầm. Dùng từ khóa raise: 

# raise một lỗi có sẵn
raise ValueError("Giá trị không hợp lệ!")

Ví dụ thực tế — kiểm tra đầu vào hàm:

def tinh_can_bac_hai(n):
    if n < 0:
        raise ValueError(f"Không thể tính căn bậc hai của số âm: {n}")
    return n ** 0.5

# OK
print(tinh_can_bac_hai(9))    # 3.0

# Tự raise lỗi
try:
    print(tinh_can_bac_hai(-4))
except ValueError as e:
    print(f"Lỗi: {e}")
# → Lỗi: Không thể tính căn bậc hai của số âm: -4

Dùng raise để re-raise lỗi sau khi log:

def doc_file(filename):
    try:
        with open(filename, "r") as f:
            return f.read()
    except FileNotFoundError as e:
        print(f"[LOG] File không tìm thấy: {filename}")
        raise   # Re-raise lỗi gốc để caller xử lý tiếp

9. Custom Exception — Tự định nghĩa lỗi

Khi xây dựng ứng dụng lớn, bạn có thể tạo exception riêng để mô tả lỗi nghiệp vụ (business logic) một cách rõ ràng hơn. Custom exception chỉ cần kế thừa từ Exception. 

# Định nghĩa custom exception
class TuoiKhongHopLeError(Exception):
    pass

class DiemKhongHopLeError(Exception):
    def __init__(self, diem, message="Điểm phải từ 0 đến 10"):
        self.diem = diem
        self.message = message
        super().__init__(f"{message}. Điểm nhận được: {diem}")


# Dùng custom exception
def kiem_tra_sinh_vien(ten, tuoi, diem):
    if tuoi < 16 or tuoi > 60:
        raise TuoiKhongHopLeError(f"Tuổi {tuoi} không hợp lệ (16–60)")
    if diem < 0 or diem > 10:
        raise DiemKhongHopLeError(diem)
    return f"{ten} | Tuổi: {tuoi} | Điểm: {diem}"


# Gọi hàm với try/except
try:
    print(kiem_tra_sinh_vien("An", 20, 9.5))    # OK
    print(kiem_tra_sinh_vien("Bình", 15, 8.0))  # → TuoiKhongHopLeError
except TuoiKhongHopLeError as e:
    print(f"Lỗi tuổi: {e}")
except DiemKhongHopLeError as e:
    print(f"Lỗi điểm: {e}")

Sơ đồ luồng thực thi của toàn bộ cấu trúc try / except / else / finally:

Tình huốngexceptelsefinally

try gặp lỗi khớp except Chạy Không chạy Chạy
try gặp lỗi KHÔNG khớp except Không chạy Không chạy Chạy 
# Ví dụ minh họa đầy đủ flow
def chia_so(a, b):
    try:
        print("→ Đang thực thi try...")
        ket_qua = a / b
    except ZeroDivisionError:
        print("→ Vào except: không thể chia cho 0")
    except TypeError:
        print("→ Vào except: a và b phải là số")
    else:
        print(f"→ Vào else: phép chia thành công, kết quả = {ket_qua}")
    finally:
        print("→ Vào finally: luôn chạy\n")

chia_so(10, 2)    # try OK → else → finally
chia_so(10, 0)    # try lỗi → except ZeroDivision → finally
chia_so(10, "x")  # try lỗi → except TypeError → finally

11. Best Practice

Luôn chỉ định loại lỗi cụ thể

# Không nên - "nuốt" mọi lỗi, khó debug
try:
    do_something()
except:
    pass

# Nên - chỉ định rõ ràng
try:
    do_something()
except ValueError as e:
    print(f"Lỗi giá trị: {e}")
except (KeyError, IndexError) as e:
    print(f"Lỗi truy cập: {e}")

Không đặt quá nhiều code vào try

# Không nên - khó biết dòng nào gây lỗi
try:
    a = int(input("Nhập a: "))
    b = int(input("Nhập b: "))
    c = a + b
    d = c * 2
    print(f"Kết quả: {d}")
    save_to_file(d)
    send_email(d)
except Exception:
    print("Có lỗi xảy ra")

# Nên - try nhỏ, chỉ bao quanh code có thể lỗi
try:
    a = int(input("Nhập a: "))
    b = int(input("Nhập b: "))
except ValueError:
    print("Vui lòng nhập số nguyên!")
else:
    c = (a + b) * 2
    print(f"Kết quả: {c}")

Dùng finally để dọn dẹp tài nguyên

# Luôn đóng file/kết nối trong finally
connection = None
try:
    connection = mo_ket_noi_db()
    du_lieu = lay_du_lieu(connection)
except DatabaseError as e:
    print(f"Lỗi DB: {e}")
finally:
    if connection:
        connection.close()   # Luôn đóng

Raise sớm, fail fast

# Kiểm tra đầu vào ngay từ đầu hàm
def tinh_toan(a, b):
    if not isinstance(a, (int, float)):
        raise TypeError(f"a phải là số, nhận được: {type(a).__name__}")
    if b == 0:
        raise ValueError("b không được bằng 0")
    return a / b

12. Ví dụ thực tế

12.1. Nhập số từ người dùng — lặp cho đến khi hợp lệ

def nhap_so_nguyen(loi_nhac="Nhập một số nguyên: "):
    while True:
        try:
            return int(input(loi_nhac))
        except ValueError:
            print("Nhập sai! Hãy nhập một số nguyên.")

so = nhap_so_nguyen("Nhập số chia: ")
print(f"Bạn đã nhập: {so}")

12.2. Đọc file an toàn

def doc_file_an_toan(duong_dan):
    try:
        with open(duong_dan, "r", encoding="utf-8") as f:
            return f.read()
    except FileNotFoundError:
        print(f"File không tồn tại: {duong_dan}")
        return None
    except PermissionError:
        print(f"Không có quyền đọc file: {duong_dan}")
        return None
    except UnicodeDecodeError:
        print(f"Lỗi encoding — thử lại với encoding khác")
        return None

noi_dung = doc_file_an_toan("data.txt")

    print(noi_dung)

12.3. Gọi API với retry

import time

def goi_api(url, so_lan_thu_lai=3):
    for lan_thu in range(1, so_lan_thu_lai + 1):
        try:
            # Giả lập gọi API
            response = requests.get(url, timeout=5)
            response.raise_for_status()
            return response.json()

        except requests.exceptions.Timeout:
            print(f"Lần {lan_thu}: Timeout. Thử lại...")
            time.sleep(2)

        except requests.exceptions.ConnectionError:
            print(f"Lần {lan_thu}: Mất kết nối. Thử lại...")
            time.sleep(2)

        except requests.exceptions.HTTPError as e:
            print(f"Lỗi HTTP {e.response.status_code}: {e}")
            break   # Không retry khi lỗi HTTP

        finally:
            print(f"  → Kết thúc lần thử {lan_thu}")

    return None

12.4. Context manager tự tạo với exception

class QuanLyFile:
    def __init__(self, ten_file, mode):
        self.ten_file = ten_file
        self.mode = mode
        self.file = None

    def __enter__(self):
        try:
            self.file = open(self.ten_file, self.mode, encoding="utf-8")
            return self.file
        except FileNotFoundError:
            raise FileNotFoundError(f"Không tìm thấy: {self.ten_file}")

    def __exit__(self, exc_type, exc_val, exc_tb):
        if self.file:
            self.file.close()
        return False   # Không suppress exception

# Sử dụng
with QuanLyFile("data.txt", "r") as f:
    print(f.read())

Bài tập 1: Đoán flow

Không chạy code, hãy dự đoán output của từng lần gọi:

def kiem_tra(x):

        print("else: không có lỗi")

        print("finally: xong")

kiem_tra(2)       # Output?
kiem_tra(0)       # Output?
kiem_tra("abc")   # Output?

Bài tập 2: Calculator an toàn

Viết hàm tinh_toan(a, phep, b) thực hiện 4 phép tính với đầy đủ xử lý lỗi. Yêu cầu:

  • Bắt ZeroDivisionError khi chia cho 0
  • Bắt TypeError khi a hoặc b không phải số
  • Dùng else để in kết quả khi thành công
  • Dùng finally để in "---" sau mỗi phép tính
def tinh_toan(a, phep, b):
    pass

tinh_toan(10, "+", 5)     # 15
tinh_toan(10, "/", 0)     # Không thể chia cho 0!
tinh_toan("x", "+", 5)   # Sai kiểu dữ liệu!

Bài tập 3: Đọc và xử lý CSV an toàn

Viết hàm doc_csv(filename) đọc một file CSV điểm sinh viên (mỗi dòng: tên,điểm). Hàm trả về list các tuple (tên, điểm). Xử lý đầy đủ: file không tồn tại, dòng bị lỗi format, điểm không phải số.

def doc_csv(filename):
    ket_qua = []
    try:
        with open(filename, "r", encoding="utf-8") as f:
            for so_dong, dong in enumerate(f, start=1):
                try:
                    ten, diem = dong.strip().split(",")
                    diem = float(diem)
                    ket_qua.append((ten.strip(), diem))
                except ValueError:
                    print(f"Bỏ qua dòng {so_dong}: format sai → '{dong.strip()}'")
    except FileNotFoundError:
        print(f"File không tồn tại: {filename}")
    return ket_qua

Bài tập 4: Custom Exception cho hệ thống điểm

Tạo custom exceptions DiemAmError và DiemQua10Error. Viết hàm them_diem(ten, diem) sử dụng hai exception đó, sau đó viết code gọi hàm với try/except để bắt từng loại.

Bài tập 5: Nhập số nguyên dương

Viết hàm nhap_so_nguyen_duong(loi_nhac) lặp cho đến khi người dùng nhập đúng một số nguyên dương (> 0). Hàm phải xử lý cả ValueError (nhập chuỗi) và trường hợp nhập số âm hoặc bằng 0.

def nhap_so_nguyen_duong(loi_nhac="Nhập số nguyên dương: "):
    while True:
        try:
            so = int(input(loi_nhac))
            if so <= 0:
                raise ValueError("Số phải lớn hơn 0")
            return so
        except ValueError as e:
            print(f"Không hợp lệ: {e}. Thử lại!")

Tổng kết

Sau bài này, bạn cần nắm chắc:

  • Exception handling giúp chương trình không crash khi gặp runtime error, cung cấp UX tốt hơn cho người dùng.
  • try: đặt code có thể gây lỗi. except: bắt buộc đi kèm try, xử lý khi có lỗi.
  • Có thể có nhiều except — Python khớp từ trên xuống, chỉ chạy cái đầu tiên khớp.
  • except LỗiCụThể as e: lưu object lỗi để in chi tiết hoặc ghi log.
  • else: chỉ chạy khi try thành công (không có lỗi).
  • finally: luôn luôn chạy — dùng để đóng file, kết nối DB, giải phóng tài nguyên.
  • raise: chủ động phát sinh lỗi khi dữ liệu không hợp lệ.
  • Custom exception: kế thừa từ Exception để tạo lỗi riêng cho ứng dụng.
  • Best practice: luôn chỉ định loại lỗi cụ thể, không để except: trống bắt hết.
